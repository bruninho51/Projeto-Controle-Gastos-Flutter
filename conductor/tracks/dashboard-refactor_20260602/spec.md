## Summary

Refatorar `dashboard_page.dart` e `main_app_scaffold.dart` seguindo princípios de Clean Architecture, SOLID e organização por feature, sem criar abstrações desnecessárias. O comportamento visual e funcional deve permanecer 100% idêntico.

## Context

O app Orçamentos já possui estrutura de features iniciada (`lib/features/`). O `dashboard_page.dart` foi recentemente movido para `lib/features/dashboard/pages/`, mas ainda concentra todo o código — chamadas de API, lógica de negócio, componentes visuais e modelos — em um único arquivo de 512 linhas. O `main_app_scaffold.dart` também mistura configuração de navegação, componentes visuais e modelos em um único lugar.

## Problemas encontrados

### Dashboard (`dashboard_page.dart`)

1. Chamadas HTTP e GraphQL dentro de `_DashboardPageState` — acesso a dados misturado com apresentação (viola SRP)
2. `_fetchOrcamentosAtivos` e `_fetchOrcamentosEncerrados` fazem **duas chamadas REST idênticas** ao mesmo endpoint — ineficiente
3. `_CardData` mistura modelo de dado com configuração visual (ícone, cor, label)
4. Configuração dos cards (ordem, ícones, cores) hardcoded na state class — difícil adicionar/remover/reordenar
5. Componentes significativos (`_DashboardHeader`, `_PillSegmentedControl`, `_DashboardCard`) são classes privadas no mesmo arquivo, sem separação física

### MainAppScaffold (`main_app_scaffold.dart`)

1. Lista `_navigationItems` definida dentro da state class — configuração de rotas misturada com lógica de navegação
2. Modelo `NavigationItem` definido no final do arquivo do scaffold, sem arquivo próprio
3. `_buildNavigationRail` (~115 linhas) e `_buildFloatingBottomNav` (~65 linhas) são métodos construtores extensos embutidos na state — são componentes com identidade própria
